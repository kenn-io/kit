# Kit v0.8.0

Kit v0.8.0 adds bounded-memory streaming across plain format-v1 pack I/O,
mixed loose/packed reads, maintenance, backup capture, verification, and
restore. It does not change the pack format or automatically raise application
storage limits.

## New capabilities

- `pack` can prepare, append, and read plain entries without whole-object heap
  allocations.
- `packstore` exposes verified streaming reads and verified copy into private
  caller staging.
- Pack, repack, and unpack stream policy-approved objects above the previous
  64 MiB practical ceiling.
- Backup capture, verification, and loose restore stream plain content.
- Active streams lease pack descriptors across cache eviction, store closure,
  and physical retirement.
- Stable pack, packstore, and backup benchmarks now have a merge-base
  regression gate suitable for reuse by external longitudinal measurement.

## Compatibility

- Pack format v1 and its footer/index meanings are unchanged.
- Packs written by Kit v0.7 remain readable, and v0.7 readers can read newly
  streamed raw and compressed format-v1 entries.
- Existing buffered and seekable APIs remain available with their previous
  all-or-nothing verification behavior.
- Existing `packstore` catalog interfaces and backup application adapters do
  not require schema changes for this upgrade.

Format-v1 encrypted entries remain buffered because their whole-frame AEAD
cannot authenticate a streamed prefix. Objects above the format's 4 GiB raw
length ceiling remain unsupported.

## Caller action

Applications may upgrade without changing policy or read paths. To receive the
bounded-memory behavior in direct content reads, move sequential consumers to
`Store.OpenStream` and treat data as authoritative only after terminal
verification. Use `Store.CopyVerified` plus private durable staging for output
that will be published.

`packstore.DefaultLimits` still caps one blob at 64 MiB. Applications should
raise that limit only after completing the
[adoption checklist](../adopting-packstore.md#adoption-checklist) with their own
catalog, archives, devices, and concurrency.

## Capacity evidence

Isolated macOS/arm64 lifecycle runs used 1 GiB raw and compressible objects.
Peak resident memory remained below 82 MiB across direct pack I/O, packstore
maintenance and mixed reads, and backup capture, verification, and restore.
Permanent CI covers frozen-reader compatibility, integrated lifecycle tests,
focused race tests, native Windows behavior, and 32-bit/unsupported-platform
builds.

These results demonstrate bounded implementation behavior; they are not a
recommended application limit. Scratch, descriptor, and aggregate memory
budgets remain workload policy.

## Release gate

Before the tag is created, at least one existing consumer must pin the exact
merged Kit commit while retaining its current object-size policy. Its normal
test suite and representative loose/packed read, write, maintenance, backup,
verify, and restore paths must pass against existing data. The release tag must
then point to that validated Kit commit; a later unvalidated documentation or
code commit is not substituted underneath the tag.
